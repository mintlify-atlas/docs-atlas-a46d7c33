In distributed messaging systems, conversation "forks" can occur when different group members have conflicting views of the conversation state. The XMTP iOS SDK includes fork recovery mechanisms to detect and resolve these conflicts.

## What is a Fork?

A fork happens when:

- Network partitions cause members to process messages in different orders
- Concurrent membership changes create conflicting group states  
- Device sync issues result in divergent conversation histories

When a fork occurs, different members may see different message histories, membership lists, or group settings.

## Fork Recovery Options

Configure fork recovery behavior using `ForkRecoveryOptions`:

```swift Client.swift:46-72
public struct ForkRecoveryOptions {
    public var enableRecoveryRequests: ForkRecoveryPolicy
    public var groupsToRequestRecovery: [String]
    public var disableRecoveryResponses: Bool?
    public var workerIntervalNs: UInt64?
    
    public init(
        enableRecoveryRequests: ForkRecoveryPolicy,
        groupsToRequestRecovery: [String],
        disableRecoveryResponses: Bool? = nil,
        workerIntervalNs: UInt64? = nil
    ) {
        self.enableRecoveryRequests = enableRecoveryRequests
        self.groupsToRequestRecovery = groupsToRequestRecovery
        self.disableRecoveryResponses = disableRecoveryResponses
        self.workerIntervalNs = workerIntervalNs
    }
}
```

### Recovery Policies

The `ForkRecoveryPolicy` enum defines when to request recovery:

## Configuration Examples

### Disable Fork Recovery

For testing or low-stakes conversations:

```swift
let options = ClientOptions(
    api: .init(env: .production),
    dbEncryptionKey: encryptionKey,
    forkRecoveryOptions: ForkRecoveryOptions(
        enableRecoveryRequests: .none,
        groupsToRequestRecovery: []
    )
)

let client = try await Client.create(account: signingKey, options: options)
```

## Advanced Configuration

### Disable Recovery Responses

Prevent this client from responding to recovery requests from others:

```swift
let options = ClientOptions(
    api: .init(env: .production),
    dbEncryptionKey: encryptionKey,
    forkRecoveryOptions: ForkRecoveryOptions(
        enableRecoveryRequests: .all,
        groupsToRequestRecovery: [],
        disableRecoveryResponses: true  // Don't help others recover
    )
)
```

<Warning>
Disabling recovery responses can prevent other group members from recovering from forks. Only use this in specialized scenarios.
</Warning>

### Custom Worker Interval

Adjust how frequently the recovery worker checks for forks:

```swift
let oneMinuteInNanoseconds: UInt64 = 60_000_000_000

let options = ClientOptions(
    api: .init(env: .production),
    dbEncryptionKey: encryptionKey,
    forkRecoveryOptions: ForkRecoveryOptions(
        enableRecoveryRequests: .all,
        groupsToRequestRecovery: [],
        workerIntervalNs: oneMinuteInNanoseconds  // Check every minute
    )
)
```

Default interval is typically 30 seconds. Longer intervals reduce network usage but increase time to detect forks.

## Detecting Forks

Check if a conversation has experienced a fork:

```swift Group.swift:531-537

let forkStatus = group.commitLogForkStatus()

switch forkStatus {
case .notForked:
    print("Conversation state is consistent")
case .forked:
    print("⚠️ Fork detected - recovery may be in progress")
case .unknown:
    print("Fork status not yet determined")
}
```

### Fork Status Values

## Recovery Process

When a fork is detected, the SDK:

1. **Detection** - Identifies that local state differs from the network
2. **Request** - Sends recovery requests to other group members
3. **Response** - Other members send their state information
4. **Resolution** - Merges states to achieve consistency
5. **Sync** - Updates local database with resolved state

### Manual Sync

Trigger a manual sync to check for and resolve forks:

```swift Group.swift:71-73
let group = try await client.conversations.findGroup(by: groupId)

```

Monitor forks across all conversations:

func checkAllGroupsForForks(client: Client) async throws {
    let groups = try await client.conversations.groups()
    var forkedGroups: [String] = []

        try await group.sync()
        
        if group.commitLogForkStatus() == .forked {

            print("Fork detected in group: \(group.id)")

    }

## Best Practices

### Sync Before Critical Operations

Sync conversations before important operations to ensure consistent state:

```swift
func sendImportantMessage(group: Group, content: String) async throws {
    // Sync first to ensure we have the latest state
    try await group.sync()
    
    // Check for forks
    guard group.commitLogForkStatus() != .forked else {
        throw MessageError.conversationForked
    }
    
    // Send message with confidence
    try await group.send(content: content)
}
```

### Handle Fork UI States

Provide clear UI feedback when forks are detected:

```swift

        }
    }
}
```

### Automatic Background Sync

Set up periodic background syncing:

```swift
class ConversationSyncManager {
    private var timer: Timer?
    private let client: Client
    
    init(client: Client) {
        self.client = client
    }
    
    func startPeriodicSync(interval: TimeInterval = 60) {
        timer = Timer.scheduledTimer(withTimeInterval: interval, repeats: true) { _ in
            Task {
                await self.syncAllConversations()
            }
        }
    }
    
    func stopPeriodicSync() {
        timer?.invalidate()
        timer = nil
    }
    
    private func syncAllConversations() async {
        do {
            let groups = try await client.conversations.groups()
            for group in groups {
                try? await group.sync()
                
                if group.commitLogForkStatus() == .forked {
                    // Log or handle forked conversations
                    print("Fork detected in group \(group.id)")
                }
            }
        } catch {
            print("Sync error: \(error)")
        }
    }
}
```

## Troubleshooting

### Persistent Forks

If forks persist after recovery attempts:

1. Ensure all group members have updated SDK versions
2. Check network connectivity - recovery requires peer communication
3. Verify recovery is not disabled on other members' clients
4. Consider leaving and rejoining the group as a last resort

### Recovery Performance

If recovery is slow:

- Reduce `workerIntervalNs` for faster detection
- Ensure adequate network bandwidth
- Check that `disableRecoveryResponses` is not set on other members
